# Правило: Генерация документа PRD (Product Requirements Document)

## Цель

Направить ИИ-ассистента на создание детального документа требований к продукту (PRD) в формате Markdown на основе первоначального запроса пользователя. PRD должен быть четким, выполнимым и понятным для младшего разработчика (Junior), чтобы тот мог реализовать функционал.

## Процесс

1. **Получение запроса:** Пользователь предоставляет краткое описание или запрос на новую функцию/возможность.
2. **Уточняющие вопросы:** Перед написанием PRD ИИ *обязан* задать только самые важные уточняющие вопросы. Ограничьтесь 3–5 критическими пробелами в понимании. Цель — понять «что» и «зачем» нужно сделать, а не «как» (это решит разработчик). Обязательно предлагайте варианты ответов в виде списков (буквы/цифры), чтобы пользователь мог легко ответить.
3. **Генерация PRD:** На основе исходного запроса и ответов пользователя создайте PRD, используя структуру, описанную ниже.
4. **Сохранение PRD:** Сохраните готовый документ под именем `prd-[название-фичи].md` в директории `/tasks`.

## Рекомендации по уточняющим вопросам

Задавайте только те вопросы, которые критически необходимы для составления ясного PRD. Фокусируйтесь на зонах, где запрос пользователя двусмыслен или не хватает контекста:

* **Проблема/Цель:** Если неясно — «Какую проблему пользователя решает эта функция?»
* **Основной функционал:** Если описание расплывчато — «Какие ключевые действия должен совершать пользователь?»
* **Границы и объем (Scope):** Если запрос слишком широк — «Есть ли что-то, чего эта функция точно *не должна* делать?»
* **Критерии успеха:** Если не указаны — «Как мы поймем, что функция реализована успешно?»

**Важно:** Задавайте вопросы только тогда, когда ответ нельзя логически вывести из исходного запроса. Отдавайте приоритет вопросам, которые существенно влияют на точность PRD.

### Требования к форматированию вопросов

* **Нумеруйте вопросы** (1, 2, 3 и т. д.).
* **Указывайте варианты ответов буквами** (A, B, C, D и т. д.) для удобства ссылки.
* Сделайте так, чтобы пользователю было просто ответить в формате «1A, 2C, 3B».

### Пример формата

```text
1. Какова основная цель этой функции?
   A. Улучшить процесс онбординга пользователей
   B. Повысить удержание (retention)
   C. Снизить нагрузку на службу поддержки
   D. Увеличить доход

2. Кто является целевым пользователем?
   A. Только новые пользователи
   B. Только существующие пользователи
   C. Все пользователи
   D. Только администраторы

```

## Структура PRD

Сгенерированный PRD должен включать следующие разделы:

1. **Введение/Обзор:** Краткое описание функции и проблемы, которую она решает. Формулировка цели.
2. **Цели:** Список конкретных, измеримых целей для данной функции.
3. **Пользовательские истории (User Stories):** Описание сценариев использования и ценности для пользователя.
4. **Функциональные требования:** Список конкретных возможностей системы. Используйте четкий язык (например: «Система должна позволять пользователю загружать фото профиля»). Пронумеруйте требования.
5. **Не-цели (Вне рамок проекта):** Четко укажите, что функция *не будет* включать, чтобы ограничить объем работ.
6. **Дизайнерские решения (Опционально):** Ссылки на макеты, описание требований к UI/UX или упоминание стилей/компонентов.
7. **Технические соображения (Опционально):** Упомяните известные технические ограничения, зависимости или советы (например: «Требуется интеграция с существующим модулем Auth»).
8. **Метрики успеха:** Как будет измеряться успех? (например: «Увеличение вовлеченности на 10%», «Снижение количества тикетов в поддержку по теме X»).
9. **Открытые вопросы:** Список оставшихся моментов, требующих дальнейшего уточнения.

## Целевая аудитория

Предполагайте, что основной читатель PRD — **Junior-разработчик**. Поэтому требования должны быть явными, недвусмысленными и по возможности без излишнего жаргона. Предоставьте достаточно деталей для понимания логики и назначения функции.

## Вывод данных

* **Формат:** Markdown (`.md`)
* **Путь:** `/tasks/`
* **Имя файла:** `prd-[название-фичи].md`

## Финальные инструкции

1. НЕ начинайте реализацию функции до завершения PRD.
2. Обязательно задайте пользователю уточняющие вопросы.
3. Используйте ответы пользователя для улучшения и детализации PRD.
