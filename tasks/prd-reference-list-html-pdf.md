# PRD: Референс-лист в современном HTML с экспортом в PDF

## 1. Введение / Обзор
Нужно преобразовать существующую референс-таблицу из файла `6. Референс-лист.docx` в современную, визуально аккуратную HTML-страницу с таблицей проектов. Итог должен быть пригоден для скачивания в формате PDF без потери структуры, читаемости и базового стиля.

Ценность задачи:
- Упорядочить и стандартизировать представление проектных данных.
- Повысить читабельность длинных записей (договоры, адреса, виды работ).
- Обеспечить единый документ для презентации/передачи (PDF).

## 2. Цели
1. Отобразить все строки и колонки из исходной таблицы `.docx` без потери данных.
2. Реализовать современный, чистый, корпоративный стиль таблицы (включая адаптивность).
3. Сделать экспорт текущего представления в PDF в 1 клик.
4. Получить PDF-файл, который можно скачать локально из интерфейса.

## 3. User Stories
- Как пользователь, я хочу открыть страницу и увидеть структурированную красивую таблицу, чтобы быстро ориентироваться по объектам и договорам.
- Как пользователь, я хочу, чтобы длинные поля корректно переносились и не ломали верстку, чтобы документ оставался читаемым.
- Как пользователь, я хочу скачать результат в PDF, чтобы передать его коллегам или заказчику.
- Как пользователь, я хочу, чтобы внешний вид PDF максимально соответствовал HTML-версии.

## 4. Функциональные требования
1. Система должна прочитать данные из подготовленного источника (массив/JSON, сформированный из `6. Референс-лист.docx`) и отрисовать их в HTML-таблице.
2. Таблица должна содержать колонки из исходного документа:
   - `№`
   - `Договор`
   - `Застройщик`
   - `Дата окончания строительства`
   - `Наименование объекта`
   - `Адрес объекта строительства`
   - `Виды работ`
   - `Краткие сведения об объекте`
3. Система должна отрисовать все строки исходной таблицы, включая записи с пустыми полями, без удаления строк.
4. Для длинных текстов система должна применять переносы, ограничение ширины колонок и удобную типографику (line-height, размер шрифта).
5. Интерфейс должен включать заголовок страницы и блок с действиями (минимум кнопка `Скачать PDF`).
6. Для стилей нужно использовать современный UI-подход:
   - Допускается использование готовых табличных паттернов/компонентов shadcn/ui.
   - Допускается Tailwind CSS или чистый CSS с дизайн-токенами.
7. Система должна поддерживать корректное отображение на desktop (обязательно) и mobile (минимум горизонтальный скролл без поломки контента).
8. Система должна экспортировать таблицу в PDF по нажатию кнопки.
9. Перед сохранением PDF система должна учитывать печатные стили (`@media print`) для лучшей читаемости.
10. Имя скачиваемого файла должно быть предсказуемым, например: `reference-list.pdf`.
11. Кодировка текста должна сохранять кириллицу в HTML и PDF.
12. Итоговый результат должен включать:
   - HTML-страницу с оформленной таблицей.
   - Возможность скачать PDF из UI.

## 5. Не-цели (Out of Scope)
- Редактирование строк таблицы пользователем через UI.
- Фильтрация, сортировка, пагинация и поиск (если не запрошено отдельно).
- Интеграция с БД/CRM/API.
- Многостраничный сайт или личный кабинет.

## 6. Дизайн-решения
- Визуальное направление: минималистичный современный B2B-стиль.
- Рекомендуемые элементы:
  - Карточка-контейнер с мягкой тенью и скруглениями.
  - Липкий заголовок таблицы (`sticky header`).
  - Зебра-строки или тонкие разделители.
  - Аккуратные отступы, контрастный текст, умеренная цветовая акцентность.
- Типографика: хорошо читаемый sans-serif, поддерживающий кириллицу.
- Если используются shadcn/ui блоки, сохранить консистентность с их токенами/паттернами.

## 7. Технические соображения
- Рекомендуемый стек:
  - `HTML + CSS + JS` (или React по необходимости).
  - Для PDF-экспорта: `html2pdf.js` или связка `html2canvas + jsPDF`.
- Важно проверить:
  - Перенос длинных значений в ячейках.
  - Масштаб и разбиение на страницы в PDF.
  - Отображение кириллицы в итоговом файле.
- Данные из `.docx` можно предварительно нормализовать в JSON для стабильного рендера.

## 8. Критерии приемки
1. На странице отображается таблица со всеми данными из `6. Референс-лист.docx`.
2. Визуальный стиль соответствует современному аккуратному оформлению.
3. Кнопка `Скачать PDF` работает и загружает PDF-файл.
4. В PDF присутствуют все строки/колонки, текст читается, кириллица не повреждена.
5. На экране шириной 360px+ страница остается функциональной (таблица доступна через горизонтальный скролл).

## 9. Метрики успеха
- 100% строк из исходной таблицы включены в HTML и PDF.
- 0 критических дефектов в отображении кириллицы.
- Успешный экспорт PDF в 100% тестовых прогонов на целевом окружении.
- Время генерации PDF до 10 секунд на стандартном ноутбуке.

## 10. Риски и ограничения
- Очень длинные поля могут ухудшать читаемость при печати в A4.
- Возможны артефакты рендера при клиентском PDF-экспорте в разных браузерах.
- Может потребоваться настройка масштаба и page-break правил для стабильного результата.

## 11. Открытые вопросы

2. Какой формат PDF обязателен: A4 portrait или landscape?
3. Допустимо ли разбиение одной строки таблицы между страницами PDF?
4. Нужны ли колонтитулы (дата формирования, номер страницы, подпись компании)?
